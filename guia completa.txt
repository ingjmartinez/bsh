GUIA COMPACTA PARA CONSTRUIR CRM V2 POR FASES
==============================================
Ultima actualizacion: 2026-05-06

Objetivo
--------

Usar esta guia para iniciar chats limpios y construir la V2 sin cargar contexto
innecesario. La regla es trabajar una sola fase por chat o por lote.

Archivos guia:

- prompt-reconstruccion-crm-v2.txt
- contexto-codigo-crm-v2.txt
- contexto-db-crm-v2.txt

Reglas fijas:

1. Una sola fase activa.
2. Leer antes de editar.
3. No inventar reglas de negocio.
4. No tocar datos productivos sin aprobacion.
5. No mezclar DB, UI, seguridad, ETL y refactor en el mismo lote.
6. No hacer Pint/formato masivo junto con cambios funcionales.
7. Cerrar cada fase con pruebas, riesgos y archivos modificados.
8. Esperar aprobacion antes de avanzar.

LAS 8 FASES RECOMENDADAS
========================

FASE 1 - Descubrimiento verificable
-----------------------------------

Objetivo:
Entender el sistema actual antes de construir.

Leer:
- composer.json
- package.json
- routes/web.php
- routes/api.php
- app/Models
- app/Http/Controllers principales
- app/Services
- app/Console/Commands
- database/migrations
- config/module_hubs.php
- config/recursos_humanos.php

Entregar:
- mapa de modulos
- entidades de dominio
- tablas criticas
- rutas GET riesgosas
- integraciones externas
- procesos/jobs/scheduler
- riesgos de DB, seguridad y performance
- modulo piloto recomendado

No editar archivos.


FASE 2 - Seguridad y rutas criticas
-----------------------------------

Objetivo:
Reducir riesgos inmediatos del sistema actual.

Incluye:
- secretos fuera del codigo
- rutas GET destructivas a POST/DELETE
- CSRF en formularios/AJAX
- permisos basicos por modulo
- dumps sensibles detectados, sin borrarlos sin aprobacion

No incluye:
- redisenar DB
- refactor masivo
- tocar tablas historicas


FASE 3 - Diseno DB V2
---------------------

Objetivo:
Disenar migraciones desde cero antes de programar modulos.

Entregar:
- ERD propuesto
- orden de migraciones
- catalogos normalizados
- FKs y politicas cascade/null/restrict
- indices principales
- estrategia tablas historicas
- decision preliminar Bet/Net
- vistas/SP/triggers a versionar

No ejecutar migraciones todavia.


FASE 4 - Modulo piloto
----------------------

Objetivo:
Construir un modulo pequeno como patron de la V2.

Candidatos:
- Tareas
- Tecnologia Solicitudes
- Entrevista Online

Debe incluir:
- migraciones limpias del modulo
- modelo con relaciones/casts/scopes
- Form Requests
- Policy
- Service/Action
- controlador delgado
- vistas/componentes
- tests minimos

No empezar por:
- ventas
- vt_usuarios
- premios
- recargas
- pagos
- incentivos completos
- BI pesado


FASE 5 - Core V2
----------------

Objetivo:
Construir la base comun del sistema.

Incluye:
- auth
- users
- roles/permisos
- hubs/modulos
- layout base
- menus desde fuente declarativa
- catalogos base
- empleados
- agencias
- rutas
- operadores/coordinadores


FASE 6 - Modulos operativos
---------------------------

Objetivo:
Construir dominios medianos ya con el patron validado.

Incluye:
- RRHH
- tecnologia
- procesos
- asistencia
- novedades
- servicios generales
- contabilidad operativa
- reporte diario rutas


FASE 7 - Incentivos y reportes
------------------------------

Objetivo:
Reconstruir reglas de incentivos y reportes sin romper negocio.

Incluye:
- CalculoIncentivo actual o reemplazo documentado
- incentivo_jobs
- incentivo_resultados
- plan agencias
- distribuciones
- efectividad
- reportes PDF/XLSX
- colas para procesos pesados

Requiere:
- reglas confirmadas
- pruebas de calculo
- validacion contra resultados legacy


FASE 8 - Datos masivos, ETL y produccion
----------------------------------------

Objetivo:
Migrar datos y preparar operacion real.

Incluye:
- ventas_producto
- vt_usuarios
- premios
- recargas
- pagos
- faltantes
- paquetico
- ventas_flash
- mar_ventas
- tablas resumen BI
- ETL por chunks
- validacion count/sum
- staging
- rollback
- backups
- health checks
- Horizon/colas
- deploy/cutover

No ejecutar sobre produccion sin aprobacion explicita.
