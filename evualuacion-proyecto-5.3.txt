============================================================
1. RESUMEN EJECUTIVO
============================================================

El proyecto es un CRM desarrollado en Laravel, con bastante funcionalidad real de negocio:
ventas, agencias, asistencia, rutas, contabilidad, incentivos, reportes, tecnologia, usuarios,
roles y permisos. No es un proyecto pequeno: tiene muchas pantallas, integraciones externas,
procesos programados y tablas de millones de registros.

Lo mejor del proyecto:
- Ya esta sobre una base moderna: Laravel 12.34.0, PHP 8.3.14, Composer 2.8.12, Livewire 3,
  Spatie Permission 6, Vite 7 y Tailwind 4 declarados.
- Tiene autenticacion, rutas protegidas por auth, modelos Eloquent, migraciones, comandos
  programados, exports Excel/PDF y separacion basica por controladores/modelos/vistas.
- La base usa modo SQL estricto con ONLY_FULL_GROUP_BY, lo cual ayuda a detectar consultas mal
  agregadas en vez de esconder errores.
- Hay indices agregados para dashboards y consultas pesadas recientes.

Lo mas urgente a corregir:
- Hay secretos y credenciales hardcodeados en codigo y docker-compose.yml. Deben rotarse y moverse
  a .env/config/services.php inmediatamente.
- Hay llamadas cURL con verificacion SSL desactivada y algunas integraciones usan HTTP o cookies
  hardcodeadas.
- Existen rutas GET que modifican o eliminan datos, por ejemplo delete-*, save-* y generar-token.
  Esto es peligroso porque un GET debe consultar, no ejecutar procesos destructivos.
- Muchas tablas productivas estan en MyISAM. MyISAM no da transacciones ni aplica llaves foraneas,
  y eso deja la integridad de datos demasiado dependiente del codigo.
- Hay una vista SQL rota/incompatible con ONLY_FULL_GROUP_BY: vw_ventas_por_producto.
- Hay migraciones pendientes.
- La suite de pruebas es practicamente de plantilla y actualmente falla.
- Frontend no es reproducible: package.json existe, pero no hay package-lock/yarn.lock/pnpm-lock y
  npm ls muestra dependencias no instaladas.
- Hay dumps SQL con datos sensibles versionados en Git.

Mi recomendacion general:
Antes de actualizar framework o versiones mayores, primero hay que asegurar secretos, rutas,
base de datos y pruebas. El proyecto tiene buena base, pero no conviene subir a Laravel 13 o PHP
8.5 sin pruebas y sin limpiar integraciones externas.

Actualizacion para trabajo con Codex/GPT-5.5:
Este documento ya no debe usarse solo como una lista de problemas. Debe funcionar como mapa de
ejecucion. GPT-5.5 es mas util cuando se le entregan tareas con frontera clara, criterio de exito,
archivos objetivo y comandos de verificacion. Por eso, las mejoras recomendadas deben ejecutarse en
lotes pequenos:
- Un lote de seguridad no debe mezclarse con formato Pint ni refactors grandes.
- Un lote de rutas debe cambiar metodos HTTP, vistas/formularios relacionados, permisos y tests del
  mismo flujo.
- Un lote de base de datos debe incluir backup, SQL exacto, validacion en staging, EXPLAIN y plan de
  rollback.
- Un lote de arquitectura debe extraer servicios sin cambiar comportamiento visible.
- Un lote de frontend debe dejar build reproducible y una prueba visual/manual clara.

Regla practica:
Cada cambio importante debe terminar con evidencia: comando ejecutado, resultado esperado, archivos
tocados y riesgo residual. Si no se puede verificar, el cambio debe quedar marcado como pendiente de
staging o pendiente de credenciales.

4. RUTAS, AUTENTICACION Y AUTORIZACION
============================================================

Resultado:
- Laravel reporta 353 rutas.
- La mayor parte del sistema esta dentro de Route::middleware('auth').
- Usuarios, roles y permisos usan middleware de Spatie Permission.
- Existe middleware role:superadmin para superadmin/sesiones.

Lo que esta bien:
- El grupo auth protege el sistema principal.
- Se usa Spatie Permission, que es una buena herramienta para RBAC.
- Los CRUD principales usan metodos HTTP correctos en varias zonas: POST, PUT/PATCH, DELETE.

Lo que esta mal:
- Hay rutas GET que ejecutan acciones de escritura o eliminacion:
  - delete-asistencias-lotobet
  - delete-faltantes-lotobet
  - delete-premios-lotobet
  - delete-recargas-lotobet
  - delete-ventas-usuarios-lotobet
  - delete-ventas-producto-lotobet
  - save-asistencias-lotobet
  - save-faltantes-lotobet
  - save-premios-lotobet
  - save-recargas-lotobet
  - save-ventas-usuarios-lotobet
  - save-ventas-producto-lotobet
  - equivalentes Lotonet y otros procesos.
- generar-token, iniciar-session y login-flash estan como GET.
- Muchas rutas de negocio solo tienen auth, pero no permisos por modulo.
- routes/api.php existe pero no se esta usando realmente; las rutas tipo api-cuentas viven en web.php
  y usan sesion web.

Por que esta mal:
- Un navegador, bot, prefetch, link compartido o imagen embebida podria disparar un GET.
- GET no tiene la misma intencion semantica de seguridad que POST/DELETE.
- Con solo auth, cualquier usuario autenticado podria tener acceso a modulos que quizas no le
  corresponden, salvo donde el controlador filtre manualmente.

Que mejorar:
- Cambiar delete-* a DELETE.
- Cambiar save-* y generar-token a POST.
- Nombrar todas las rutas criticas.
- Aplicar middleware permission por modulo: ventas.sync, ventas.delete, agencias.manage,
  contabilidad.manage, incentivos.manage, tokens.generate, etc.
- Agregar confirmaciones de UI, logs de auditoria y rate limiting.
- Separar rutas API reales en routes/api.php con auth:sanctum o token interno si se exponen fuera.

Hallazgos criticos:
- docker-compose.yml contiene credenciales de base de datos hardcodeadas.
- app\Http\Controllers\TokenController.php contiene endpoints, headers, usuario/password/cookie

- app\Http\Controllers\Api.php contiene tokens de API externa hardcodeados.
- app\Http\Controllers\AsistenciaController.php y otros controladores tienen tokens/cookies/credenciales
  externas hardcodeadas.
- Varias llamadas cURL usan:
  - CURLOPT_SSL_VERIFYHOST => 0
  - CURLOPT_SSL_VERIFYPEER => 0
- Hay dumps SQL versionados con datos personales y tokens.

Por que esta mal:
- Si el repositorio se comparte o se sube a un remoto, las credenciales quedan expuestas.
- Desactivar verificacion SSL permite ataques man-in-the-middle.
- Los dumps SQL contienen informacion sensible de empleados/clientes/agencias y no deben vivir en Git
  sin sanitizar.

Que esta bien:
- .env esta en .gitignore.
- AuthController usa Hash para password y Auth::attempt.
- Hay rate limit manual para reset de contrasena por correo.
- Hay CSRF en formularios POST por el middleware web.

Que mejorar de inmediato:
1. Rotar todas las credenciales que aparecen en codigo, dumps o docker-compose.yml.
2. Mover secretos a .env y config/services.php.
3. Usar Laravel HTTP Client en vez de cURL crudo donde sea posible.
4. Activar verificacion SSL.
5. Eliminar cookies hardcodeadas y manejar login/token con expiracion real.
6. Sacar dumps sensibles del repositorio y reescribir historia si ya fueron compartidos.
7. Agregar .gitignore para database/*.sql, database/*.xlsx, database/*.csv si son datos reales.
8. Usar variables de entorno en Docker con env_file o secretos del orquestador.

Observacion importante:
Aunque las migraciones declaran varias llaves foraneas, muchas tablas terminaron en MyISAM. MyISAM no
aplica llaves foraneas y no soporta transacciones como InnoDB. Por eso information_schema solo muestra
5 tablas con FKs reales. Esto es uno de los mayores riesgos del sistema.

Vista rota:
- vw_ventas_por_producto falla con ONLY_FULL_GROUP_BY.
- El error ocurre porque selecciona v.monto como total_ventas sin agregarlo ni incluirlo en GROUP BY.
- Tambien aparece sum(c.tipo), que conceptualmente parece incorrecto porque tipo normalmente es texto
  o categoria, no un campo numerico para sumar.

Por que esto importa:
- php artisan db:show --counts --views falla al intentar contar esa vista.
- Los dashboards o reportes que dependan de esa vista pueden fallar en servidores con SQL estricto.
- Si se apaga ONLY_FULL_GROUP_BY para "arreglarlo", se esconderia el problema y se obtendrian resultados
  ambiguos.

Que mejorar en DB:
1. Antes de tocar estructura: backup completo y prueba en staging.
2. Convertir tablas MyISAM a InnoDB gradualmente:
   ALTER TABLE nombre_tabla ENGINE=InnoDB;
3. Agregar primary keys a tablas sin PK, empezando por ventas_producto_net y tokens.
4. Revisar tipos de columnas de relacion: agencia_id, terminal, producto_id, fecha, cedula.
5. Normalizar criterios de terminal/agencia para no depender tanto de TRIM/CAST en joins.
6. Agregar indices compuestos segun reportes reales:
   - fecha, agencia_id
   - fecha, producto_id
   - agencia_id, fecha
   - producto_id, fecha
   - cedula, fecha
7. Corregir vw_ventas_por_producto:
   - usar SUM(v.monto) AS total_ventas
   - agrupar por producto_id, descripcion, tipo
   - eliminar sum(c.tipo)
8. Evaluar particionamiento o tablas resumen mensuales para ventas/premios/recargas.
9. Crear jobs de agregacion diaria para dashboards en vez de recalcular sobre 50M+ filas.
10. Mantener ONLY_FULL_GROUP_BY activado.

15. STACK RECOMENDADO 2026

Recomendacion conservadora para tu proyecto:
- PHP: 8.4 en produccion a corto plazo; 8.5 despues de pruebas.
- Laravel: mantener 12 por ahora; planificar 13 despues de limpiar seguridad y tests.
- MySQL: 8.4 LTS para produccion estable.

- Node: 24 LTS para build.
- Frontend: Vite + Tailwind 4, pero con package-lock.json versionado.
- Testing: PHPUnit 11 mientras estes en Laravel 12; PHPUnit 12 al subir a Laravel 13.
- Calidad: Laravel Pint obligatorio en CI.
- Analisis estatico opcional recomendado: Larastan/PHPStan.
- Deploy: Docker con imagen inmutable, sin bind mount del codigo en produccion.

Por que no recomiendo saltar directo a Laravel 13 hoy:
- La app tiene rutas GET destructivas.
- Hay secretos hardcodeados.
- Hay tests fallando y casi sin cobertura.
- Hay vista SQL rota con ONLY_FULL_GROUP_BY.
- Hay tablas MyISAM y sin PK.
- Hay controladores/vistas muy grandes.

Actualizar framework sin resolver eso puede convertir problemas existentes en fallas mas dificiles
de depurar.

18. DIAGNOSTICO FINAL
============================================================

El proyecto esta vivo y tiene una base tecnica moderna, pero crecio rapido y ahora necesita una ronda
seria de endurecimiento. No lo veo como un proyecto malo; lo veo como un sistema de negocio que ya
resuelve problemas reales, pero que arrastra deuda normal de crecimiento: integraciones directas,
controladores grandes, SQL pesado, datos sensibles en repo y poca cobertura.

Prioridad real:
1. Seguridad de secretos y rutas.
2. Base de datos: InnoDB, PK, FKs, vista rota, indices y tablas resumen.
3. Pruebas automaticas.
4. Separacion de servicios/jobs.
5. Frontend reproducible.
6. Luego si: actualizar Laravel/PHP/MySQL con confianza.

Conclusion actualizada:
El proyecto no necesita una reconstruccion total antes de avanzar; necesita una actualizacion
ordenada. La mejor ruta es endurecer primero lo que puede exponer datos o romper operacion, despues
crear pruebas minimas, y luego modernizar arquitectura por modulo. Con GPT-5.5/Codex conviene trabajar
en cambios pequenos y verificables, no en refactors masivos. Si se sigue ese orden, actualizar el
stack deja de ser una apuesta y se convierte en un proceso controlado.

Primeros 5 trabajos recomendados para el siguiente ciclo:
1. Sacar secretos de TokenController, Api.php, AsistenciaController y docker-compose.yml.
2. Convertir las rutas GET destructivas de un solo modulo a POST/DELETE con permisos y pruebas.
3. Corregir vw_ventas_por_producto y validar totales en staging.
4. Arreglar ExampleTest y agregar una prueba real de login/permisos.
5. Generar package-lock.json y confirmar que npm run build funciona.
